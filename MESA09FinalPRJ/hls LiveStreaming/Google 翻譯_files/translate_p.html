我一直熱衷於流媒體視頻，網絡攝像機等，並且一段時間以來試圖找出向我的iPhone提供實況/近乎實況視頻的最佳方式（免費）。 I've always been kind of keen on streaming video, webcams and so on and have for some time tried to work out the best way of delivering a live/almost live video feed to my iPhone (for free that is).

昨天晚上，我終於在搜索網絡的過程中獲得了突破。 Last night I finally got a breakthrough while searching the web for possibilities. 我有一段時間有一個純粹的HTML5流使用VLC編碼的Theora / OGG運行，但這只能在谷歌瀏覽器中看到。 I had for some time had a pure HTML5 stream running using Theora/OGG encoded with VLC, but this was only viewable with Google Chrome.

HTTP Live Streaming基本上是Apple和其他一些公司用來從3G傳輸視頻內容到WI-FI連接的技巧。 HTTP Live Streaming is basically a trick used by Apple and a few others to deliver video content over anything from 3G to WI-FI connections. 它的工作原理是將視頻分成小塊，然後逐一將它們下載到客戶端，然後再將它們放回到一起。 It works by splitting a video into small chunks and then downloading them one by one to the client which then puts them back together. 要知道要下載哪些位，該協議使用的索引文件實際上只是一個URL列表。 To know which bits to download, the protocol makes use of an index file which is really just a list of URLs. 所有這些對於靜態內容都非常簡單 - 只需將您的視頻轉碼為正確的格式（MPEG2容器中的H264）並將其剪下即可。 All this is pretty easy with static content – just transcode your video into the correct format (H264 in an MPEG2 container) and chop it up. 然後提供一個索引文件（.m3u8）並將它指向它。 Then provide an index file (.m3u8) and point your device at it. 那裡有軟件可以為你做到這一點。 There is software out there that can do this for you.

但對於實況視頻來說，這是一個挑戰。 But for live video there's a challenge. 首先，沒有固定的視頻長度 - 它會一直持續下去。 First of all, there is no set length of the video – it just goes on and on. 那麼大多數HTTP Live Streaming軟件所做的就是將視頻源以塊形式存儲，就像使用靜態視頻一樣，然後動態更新索引文件以指向最後說的5個塊。 So what most of the HTTP Live Streaming software out there does is that it stores the video feed in chunks, just like with a static video, and then dynamically updates the index file to point to the last say 5 chunks. 客戶端不斷下載索引文件，所以它總是知道要下載並下一個播放的塊的名稱。 The client continuously downloads the index file, so it always knows the name of the chunk to download and play next. 這種方法的主要缺點是客戶端看到的內容並不真實，而是具有<塊大小*塊數>的延遲 - 通常約為60秒。 The main drawback of this method is that the content the client sees is not really live, but instead has a latency of <chunk size * number of chunks> – usually about 60 seconds.
